The NAIT Skill

How Bookmarked ensures every automation, workflow, and AI task starts with a high-fidelity specification — before a single line of work gets executed.

The One-Liner

NAIT (Nate B. Jones — Ops Architect) is Bookmarked's upstream quality gate. It forces every workflow request through a structured specification pass — extracting acceptance criteria, classifying the right human/AI boundary, mapping context dependencies, and designing the handoff card — before any execution skill touches it. The result: prompts that are complete, bounded, and testable from day one.

Why This Skill Exists

Most AI failures aren't intelligence failures — they're specification failures. An agent given a vague request will produce vague output. An agent given a precise spec with acceptance criteria, context pointers, and a clear handoff will produce work that lands the first time.

The core insight: 60 seconds of structured specification upstream saves hours of rework, misalignment, and "that's not what I meant" downstream. NAIT enforces that discipline as a non-negotiable gate.

Without NAIT

  • "Build me an onboarding flow" → agent guesses scope, builds the wrong thing
  • No acceptance criteria → no way to know if the output succeeded
  • No handoff design → unclear where AI stops and human starts
  • Missing context → agent reinvents what already exists in Confluence
  • Rework loops burn team trust in the system

With NAIT

  • "Build me an onboarding flow" → spec-checked with 3 testable criteria first
  • Acceptance criteria gate enforced → every output is measurable
  • Handoff card designed → agent and human roles are explicit
  • Context mapped → agent loads exactly the right pages/files
  • Execution skills consume specs without rework

The Four-Step Single-Pass Workflow

NAIT runs one continuous pass through four steps. No phase gates, no back-and-forth — one response, complete spec. Takes roughly 60 seconds.

1

Problem + Acceptance Criteria

Extracts the trigger, desired outcome, and stakeholder. Then drafts exactly 3 acceptance criteria:

  • Success condition — The measurable proof the workflow worked
  • Boundary condition — The proof the workflow did not overreach
  • Constraint — The time, format, or resource limit
Hard gate: If someone says "just build it" without criteria, NAIT refuses and drafts candidate criteria from their description instead. This is the only non-negotiable gate in the entire skill.
2

Context Pointers

Identifies which Confluence pages and repo files the execution skill will need. Flags gaps ("you'll need a page for X — it doesn't exist yet"). Validates the workflow against the Intent Manifest's trade-off hierarchy and hard boundaries.

3

Handoff Card

Defines the precise boundary between AI and human responsibility:

Claude deliversSpecific artifact or action the AI produces
Human verifiesWhat requires human judgment — tone, strategic fit, relationship sensitivity
Claude self-checksWhat the AI validates before handoff — fields populated, format correct, saved correctly
Progress stateWhere state lives if multi-session (session plugin, MEMORY.md, Confluence)
4

Execution Routing

Identifies which existing skill(s) should execute the spec, estimates session count, and lists dependencies (API access, human inputs, data availability, external coordination).

The Workflow Classification Engine

Every workflow NAIT processes gets classified using five diagnostic questions. This classification determines the right human/AI boundary — not a guess, but a structured assessment.

Question If Yes If Partial If No
Q1: Can the decision logic be fully encoded? Agent-Ready Agent-Augmented Human-Only
Q2: Is the outcome reversible if AI errs? Agent-Ready Agent-Augmented Human-Only
Q3: Can success be measured without human interpretation? Agent-Ready Agent-Augmented Agent-Augmented at best
Q4: Is this internal (no external stakeholder expectations)? Agent-Ready Agent-Augmented Human-Only
Q5: Would a 2 AM agent produce the same quality as a human at 2 PM? Agent-Ready Agent-Augmented Human-Only

Agent-Ready

AI executes end-to-end. Human sets boundaries and reviews occasionally.

Agent-Augmented

AI does the heavy lifting. Human reviews at a defined gate before it ships.

Human-Only

AI prepares materials. Human leads, decides, and executes.

Classifications are not permanent. As decision boundaries mature and feedback loops prove alignment, workflows can be promoted. Demotions happen when quality degrades or new edge cases emerge.

The Expertise NAIT Brings

Specification Rigor

Forces every request through structured decomposition. Vague requests become testable specs. "Build me a thing" becomes trigger + outcome + 3 acceptance criteria.

Acceptance Criteria Enforcement

The only non-negotiable gate: no workflow spec ships without 3 acceptance criteria (success, boundary, constraint). Refuses to proceed without them.

Human/AI Boundary Design

Uses the 5 Classification Questions to determine exactly where AI should stop and human judgment should start. No guessing — structured diagnostic.

Context Dependency Mapping

Identifies Confluence pages and repo files the execution skill will need. Flags gaps before work begins, not after.

Intent Alignment Validation

Every spec is validated against the Intent Manifest's trade-off hierarchy and hard boundaries. Workflows that conflict with organizational intent get flagged immediately.

Execution Routing

Identifies which existing skills should execute the spec, estimates session count, and lists dependencies — so execution starts without discovery overhead.

How This Helps the Team

  • Higher first-pass quality: When the spec is complete, execution skills produce work that lands the first time instead of requiring multiple revision cycles.
  • Consistent AI behavior: Every team member who runs a workflow through NAIT gets the same structured specification process, regardless of how they phrase their request.
  • Clear accountability: The handoff card makes it explicit what AI is responsible for and what the human is responsible for. No ambiguity about who owns what.
  • Faster onboarding: New team members can see exactly what a well-specified prompt looks like — the output IS the training material.
  • Reduced rework: Acceptance criteria defined upfront means you know if the output succeeded before it ships. No more "that's not what I meant."
  • Organizational memory: Context pointers ensure execution skills load existing organizational knowledge instead of reinventing it.
  • Audit trail: Every workflow spec is saved to claude-output/exports/ — the team can review what was specified, what was executed, and whether they matched.

The Decision Boundaries

NAIT operates at Autonomy Level 3 (Advisor) — it recommends, you decide.

Can Autonomously

  • Run the single-pass spec-check
  • Search Confluence for context
  • Draft problem statements, criteria, and handoff cards
  • Classify workflows using the 5 questions
  • Audit existing specs for compliance

Must Escalate When

  • Workflow involves external stakeholders (customers, partners, investors)
  • Proposed workflow overlaps >30% with an existing skill
  • Workflow requires creating new Confluence pages

Never Does

  • Produces a spec without acceptance criteria
  • Executes workflows — design only
  • Skips the handoff card
  • Creates Confluence pages directly
  • Duplicates referenced framework content

Trade-Offs Authorized

  • Rigor over speed: building the wrong thing costs more than 60 seconds of spec work
  • Proportional context: a 1-step update doesn't need 5 Confluence pages mapped

Review Mode

NAIT also has a review mode (/nate:review) that audits existing workflows, session plugins, or specs for compliance. It checks for each required element and scores them as present, missing, or incomplete — then drafts what's missing.

Element Checked What It Catches
Problem statement Workflows without a clear trigger, outcome, or stakeholder
Acceptance criteria Workflows with no way to measure success or detect overreach
Classification Workflows misclassified as Agent-Ready when they need human gates
Context pointers Workflows missing required Confluence or repo references
Handoff card Workflows with unclear AI/human boundaries
Execution routing Workflows with no clear path to implementation
