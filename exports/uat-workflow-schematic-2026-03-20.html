Iteration Loops

The workflow above runs per-sprint. But three feedback loops run at different cadences to improve the process itself — not just the product.

Every 1–2 Days (Per Sprint)

Sprint Retro

After each sprint's UAT completes, the system learns what broke and why.

  • 1.Agent generates sprint UAT summary (pass rates, failure patterns, time to test) Agent
  • 2.Sarah reviews: Were any acceptance criteria missing that caused false passes? Were priorities correct? Sarah
  • 3.If acceptance criteria quality was low, Sarah updates the template or trains the /jira skill to catch it next time Sarah
  • 4.If a test couldn't run (missing credentials, wrong environment), file an infrastructure ticket Sarah
Weekly (Thursday 1:1)

Workflow Refinement

Steve and Sarah review how well the agents are working — not just the product, but the process.

  • 1.Agent pulls trend report: pass rates by persona over last 5 sprints, stuck bugs, regression frequency Agent
  • 2.Steve + Sarah review: What's the /uat skill getting wrong? What's it missing? What should it do differently? Steve Sarah
  • 3.Sarah documents needed skill changes (new test scenarios, updated prompt logic, additional checks) Sarah
  • 4.Steve builds the skill updates based on Sarah's specs Steve
  • 5.Cross-persona patterns escalated to Raju for systemic fixes (e.g., campus context bug = 24 files) Raju
Quarterly

Process Evolution

Step back and ask: Is this workflow still right? What's changed about our product, team, or tools?

  • 1.Agent generates quarterly quality report: defect trends, coverage gaps, personas never tested, time-to-fix metrics Agent
  • 2.Are we testing the right things? New features may need new inventory items or new personas Sarah
  • 3.Does the /uat skill need new capabilities? (e.g., performance testing, accessibility, mobile viewport) Steve
  • 4.Are the Feature Inventory pages still accurate? Retire deprecated features, add new ones Sarah
  • 5.Update this schematic if the workflow itself has changed Steve
Who Owns What

Clear ownership prevents gaps and prevents duplication. The principle: humans define and decide, agents execute and report.

What We Stop Doing

Each tester writes their own prompt → One prompt, baked into /uat skill, everyone gets the same test
Editing Feature Inventory page body with results → Results go as Confluence comments only — page body is the source of truth for what to test
Manually creating Jira tickets from test failures → Agent creates bugs automatically with evidence, human reviews and prioritizes
Testing with no cross-persona checks → Every test includes interference checks for adjacent personas
Steve builds everything, team reacts → Sarah defines process changes, Steve builds them into agents, agents execute
No visibility between 1:1s → Daily async pulse (Slack) + agent-generated sprint summaries
